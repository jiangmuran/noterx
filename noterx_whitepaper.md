# 薯医 / NoteRx

## 项目一句话介绍

**AI驱动的小红书笔记诊断平台——用数据告诉你，你的笔记为什么没火。**

---

## 一、命名方案

### 首选：（NoteRx）

- "薯"= 小红薯（小红书社区昵称），"医"= 诊断、处方、治疗
- Rx 是医学处方符号，NoteRx = 给你的笔记开处方
- 四个字，过目不忘，自带产品定位
- Slogan：**「你的笔记，值得被看见。」**

## 三、核心卖点（对齐评审维度）

### 卖点1：真实痛点，真金白银

小红书上有超过8000万创作者。绝大多数素人博主发了笔记没有流量，但不知道问题出在哪——是封面不行？标题不吸引？标签选错了？发布时间不对？目前没有任何工具能给出系统性的诊断。

**薯医就是这个工具。**

→ 对应评审维度：**商业潜力**

### 卖点2：不是玄学，是数据科学

我们不是让AI随便给建议。我们针对2-3个垂类（美食/穿搭/科技），提前采集数千条真实小红书笔记数据，建立**量化评价基线（Baseline）**：

- 该垂类爆款笔记的标题平均字数、关键词分布
- 封面构图的色彩饱和度、文字占比、人脸出现率
- 标签使用策略与互动率的相关性
- 发布时间与曝光量的关系
- 博主身份粉丝量

用户的笔记不是和"AI的感觉"对比，是和**真实数据**对比。这个数据体系就是我们的护城河。

→ 对应评审维度：**技术难度**

### 卖点3：多Agent辩论，不是一言堂

单个AI给建议容易片面。薯医采用**多Agent架构**：

- **内容分析师Agent**：分析文案结构、信息密度、可读性
- **视觉诊断Agent**：分析封面构图、配色、视觉吸引力
- **增长策略Agent**：分析标签、发布时间、互动策略
- **用户模拟Agent**：模拟目标受众的第一反应（融合AI评论区概念）
- **综合裁判Agent**：汇总各Agent观点，处理分歧，给出最终诊断报告

Agent之间会互相质疑——比如视觉Agent说封面很好，但用户模拟Agent说目标用户其实不喜欢这种风格——这种辩论过程本身就是产品亮点，可以展示给用户看。

→ 对应评审维度：**创新性 + 技术难度**

### 卖点4：诊断报告天然是小红书内容

薯医生成的诊断报告是一张精心设计的**诊断卡片**——包含评分雷达图、问题标注、优化建议——用户拿到这张卡就会想发到小红书上："我的笔记被AI诊断了，原来问题出在这里！"

这和PlanCoach的传播逻辑一模一样：产品输出物本身就是社交货币。

→ 对应评审维度：**商业潜力（增长飞轮）**

### 卖点5：我们自己就是用户

队员陈宇夏是自媒体博主，真实运营小红书账号。这不是一群程序员在想象用户需求，这是创作者在解决自己的问题。

→ 对应评审维度：**现场讲述效果**

## 四、产品功能拆解

### 核心流程

```
用户上传笔记（截图/文字/链接）
         ↓
    多模态内容解析
  （OCR + 图像分析 + NLP）
         ↓
   与垂类Baseline数据对比
         ↓
    多Agent并行诊断
  （内容/视觉/增长/用户模拟）
         ↓
    Agent辩论 & 综合裁判
         ↓
   生成诊断报告 + 优化建议
         ↓
   一键生成优化版笔记（可选）
         ↓
   导出诊断卡片（可分享）
```

### 功能模块

| 模块 | 说明 | 优先级 |
|------|------|--------|
| 笔记上传 | 支持截图、粘贴文字、输入链接 | P0 |
| 封面诊断 | 构图分析、色彩评分、文字占比、人脸检测 | P0 |
| 标题诊断 | 关键词吸引力、长度、情绪词、与垂类爆款对比 | P0 |
| 内容诊断 | 结构完整性、信息密度、可读性评分 | P0 |
| 标签诊断 | 标签数量、热度匹配、长尾策略 | P1 |
| 用户预测 | AI模拟评论区反应（融合AI评论区） | P1 |
| Agent辩论可视化 | 展示多Agent讨论过程 | P1 |
| 一键优化 | AI重写标题/重排内容/推荐封面方向 | P2 |
| 诊断卡片导出 | 生成可分享的诊断结果图 | P0 |

## 五、技术架构

```
┌─────────────────────────────────────────────┐
│                 前端（Web App）                │
│  上传界面 → 诊断动画 → 报告展示 → 卡片导出     │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│              API Gateway                     │
└──────────────────┬──────────────────────────┘
                   │
        ┌──────────┼──────────┐
        ▼          ▼          ▼
  ┌──────────┐ ┌────────┐ ┌──────────┐
  │ OCR/图像 │ │  NLP   │ │ 数据对比 │
  │  分析层  │ │ 分析层 │ │   引擎   │
  └────┬─────┘ └───┬────┘ └────┬─────┘
       └───────────┼───────────┘
                   ▼
        ┌─────────────────────┐
        │  多Agent编排引擎     │
        │                     │
        │  内容Agent ←→ 视觉Agent  │
        │      ↕           ↕      │
        │  增长Agent ←→ 用户Agent  │
        │         ↘   ↙          │
        │      综合裁判Agent      │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │   诊断报告生成器     │
        │  雷达图 + 建议 + 卡片 │
        └─────────────────────┘

┌─────────────────────────────────────────────┐
│          本地数据层（Baseline）               │
│  美食垂类×2000条  穿搭垂类×2000条  科技×2000条 │
│  预计算统计指标：标题词频、封面特征、互动率分布   │
└─────────────────────────────────────────────┘
```

### 技术栈建议

- **前端**：React + Tailwind（前期demo搭建框架用native js），做成Web App（不做小程序，48小时来不及审核）
- **AI后端**：Python FastAPI
- **LLM**：Claude API / GPT-4o（多Agent编排）
- **图像分析**：OpenCV（封面构图）+ CLIP（视觉语义）
- **数据层**：SQLite本地存储baseline数据，提前采集清洗好带进赛场
- **图表**：ECharts生成雷达图
- **卡片导出**：html2canvas截图导出

## 七、路演策略（EXPO + 决赛）

### EXPO阶段（展览会，需要在人群中抓住评委）

**30秒pitch：**

> "你发过小红书吗？有没有觉得明明内容很好但就是没人看？薯医用AI帮你诊断问题出在哪——不是玄学，是基于几千条真实数据的科学分析。来，把你最近发的笔记截图给我，我现场帮你看。"

**关键动作：让评委现场体验。** 准备一台iPad，评委扫码就能用。如果评委自己有小红书账号，直接诊断他的笔记——这个互动感没有任何PPT能替代。

### 决赛阶段（8分钟路演）

**结构：**

| 时间 | 内容 | 要点 |
|------|------|------|
| 0-1min | **痛点故事** | 陈宇夏讲自己作为博主的真实经历："我发了50篇笔记，有的爆了有的石沉大海，我不知道为什么。" |
| 1-2min | **产品演示** | 现场把陈宇夏一篇没火的笔记丢进薯医，实时生成诊断 |
| 2-4min | **技术深度** | 展示多Agent辩论过程、baseline数据体系、量化评价模型 |
| 4-5min | **用户预测** | 展示AI模拟评论区功能——"这是AI预测的用户反应" |
| 5-6min | **数据验证** | 用一篇已知爆款笔记反向验证：薯医的诊断和真实数据吻合 |
| 6-7min | **商业故事** | 创作者工具SaaS模型、垂类扩展路径、与小红书生态的协同 |
| 7-8min | **团队故事** | "我们是全场最年轻的队伍。我们不是在想象用户，我们就是用户。" |

### 杀手锏

路演最后，把**薯医自己的参赛笔记**（提交要求里要发的那篇小红书笔记）丢进薯医做诊断。用自己的产品诊断自己的产品——这个meta效果会让全场记住你们。

## 十、风险与应对

| 风险 | 概率 | 应对方案 |
|------|------|---------|
| baseline数据来不及采集 | 中 | 赛前就开始采，只做2个垂类也够demo |
| AI诊断结果不准确 | 中 | 展示Agent辩论过程而非单一结论；标注"AI建议仅供参考" |
